# 跨境連線 POR 系統設計建議方案

為了克服中國 GFW 對於傳統 VPN (如 Tailscale/WireGuard) 的封鎖工程，我們需要更具「抗封鎖性」或「架構冗餘」的方案。

## 最佳化建議方案：中方主動推送鏡像 (Reverse Data Sync)
既然中國端沒有公網 IP，且可接受 1 分鐘延遲，最穩定的方法是讓**大陸端主動「推」數據給台灣端**。

### 1. 網路拓撲
- **台灣端 (Server)**：
    - 需要一個公網 IP (或是家用寬頻開啟 Port Forwarding)。
    - 運行一個極簡的「接收端 API」 (Sync Receiver)。
- **中國端 (Client)**：
    - POR 系統照常運行。
    - 運行一個「同步腳本」，每 60 秒檢查 `.sqlite` 檔案，若有變動則主動傳輸。

### 2. 優點
- **無需認證**：因為數據是從中國流向台灣（且通過加密隧道或私有 API），不涉及在中國境建立對公眾開放的網站，因此**不需要 ICP 備案或站點認證**。
- **穿透力強**：大陸主動發起對台灣的連線，能跳過大部分針對入站 (Inbound) 的封鎖。
- **高可用性**：即便連線暫時中斷，台灣端的樹莓派因為跑的是本地 Docker + 本地 SQLite 副本，看板依然會亮著。

### 3. 技術實作建議 (三步驟)

#### 第一步：台灣端設置「接收港」(Sync Receiver)
在台灣樹莓派建立一個極簡的 Flask API (或使用 `rsync` 服務)，專門接收來自中國的資料庫。
- **安全性**：設定 API Token 認證，只允許持有密鑰的大陸終端上傳。
- **自動刷新**：接收到新資料庫後，台灣端的 POR 服務會因資料庫更新而反映最新看板。

#### 第二步：中國端設置「推送員」(Sync Pusher)
在大陸端運行一個 Python 監控腳本：
- **檢測變動**：監控 `instance/projection.sqlite` 的最後修改時間。
- **主動傳輸**：當數據變動且距離上次上傳超過 60 秒時，透過加密通道 (HTTPS 或隧道) 將檔案推送到台灣。

#### 第三步：建立加密隧道 (抗封鎖保障)
若發現 HTTP(S) 直接傳送會被干擾，則在大陸端部署 **Hysteria2 客戶端**，並將推送腳本的流量導向 Hysteria2 的 SOCKS5 代理。這將使數據傳輸完全偽裝成常見的網頁流量。

## 方案 C：雲端企業級加速 (穩定付費型)
- **運作模式**：使用阿里雲 Global Accelerator (GA) 或類似的跨境加速服務。
- **優點**：企業級穩定，低延遲，免維護。
- **缺點**：成本較高。
