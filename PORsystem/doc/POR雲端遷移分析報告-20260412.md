# POR 系統遷移至雲端/網路主機可行性分析報告

## 1. 總結 (Executive Summary)
將 POR 系統部署在公網主機是 **技術上可行** 的，因為其核心架構基於標準的 Python Flask 與 SQLite，具備良好的跨平台相容性。然而，由於現有系統是針對樹莓派（Raspberry Pi）邊緣運算場景設計，若要遷移至雲端，需要解決 **遠端硬體控制失效**、**安全性暴露** 以及 **網路穩定性依賴** 三大核心問題。

---

## 2. 潛在問題與挑戰

### A. 遠端硬體控制失效 (Critical)
*   **現象**：管理介面中的「重啟看板」與「關閉系統」功能是直接調用 `os.system('sudo reboot')`。
*   **風險**：部署在網路主機時，點擊這些按鈕會導致 **雲端伺服器重啟/關機**，而非顯示看板的螢幕硬體。
*   **建議**：需移除或停用管理介面的硬體控制按鍵，或改為與現場播放器（Client）通訊的指令。

### B. 安全性與隱私風險 (Important)
*   **現象**：目前系統依靠內網 IP 進行管理權限初步判定，且未強制執行複雜的帳密登入。
*   **風險**：一旦暴露在公網，任何知道網址的人都能修改看板內容甚至刪除數據。
*   **建議**：必須實施具備 Session 或 JWT 機制的身份驗證系統，並啟用 HTTPS 編碼通訊。

### C. 網絡依賴性與穩定性
*   **現象**：系統採用 5 秒一次的智能輪詢（Smart Polling）進行數據同步。
*   **風險**：
    *   **斷網即失效**：不同於本地部署，若現場網路中斷，網頁瀏覽器將無法載入後端數據，看板會顯示錯誤白畫面。
    *   **延遲累積**：公網延遲可能使「5 秒同步」的體驗下降。
*   **建議**：考慮實施 Service Worker 或 PWA 技術進行離線緩存，確保基本內容在斷網時仍能顯示。

---

## 3. 技術調整方案建議

| 項目 | 樹莓派部署 (現狀) | 網路主機部署 (計畫) | 建議調整 |
| :--- | :--- | :--- | :--- |
| **資料庫** | 本地 SQLite | 雲端 SQLite 或 PostgreSQL | 若流量大建議遷移至 PostgreSQL |
| **硬體重啟** | 可透過 Web 觸發硬體重啟 | **無法觸發現場硬體重啟** | 改為現場定時重啟腳本 |
| **身分驗證** | 基於 IP 或簡易 Token | **必須強化帳密與權限管理** | 遷移至 Role-based Auth |
| **Kiosk 模式** | 指向 `127.0.0.1` | 指向 `https://your-domain.com` | 修改 Chrome 啟動腳本 (.bat) |
| **維護成本** | 一次性硬體支出 | 每月主機租借費用 | 評估長期運營成本 |

---

## 4. 下一步行動建議
如果您決定執行遷移，我建議分階段處理：
1.  **環境適應性修改**：修改 `main.py` 以區分雲端與本地環境，停用危險的 `os.system` 指令。
2.  **安全強化**：實施登入介面，確保管理端安全。
3.  **現場播放器配置**：現場的電視仍需要一台小主機（如原本的樹莓派或迷你 PC）僅作為「瀏覽器端」，開機自動進入 Kiosk 模式導向雲端網址。
