# POR 系統 VPS 雲端部署與無損更新策略

為了確保在 VPS 運行過程中新增功能且不遺失任何客戶資料，建議採用以下 **「資料與代碼分離」** 的標準運維流程：

## 1. 核心原理：Docker 數據持久化 (Isolation)
目前的系統架構已經將「代碼」與「資料」完全分離。資料儲存在宿主機的實體資料夾中，透過 Docker 的 `Volumes` 掛載：
- `database/`: 存放 SQLite 資料庫。
- `instance/`: 存放操作日誌。
- `config.inf`: 存放所有權限與佈景設定。

**更新優勢**：當您拉取新程式碼並重啟容器時，Docker 僅更換「運算引擎」，這些實體檔案會保持不動。

---

## 2. 建議的 VPS 更新工作流 (Standard Workflow)

### 第一步：本地驗證 (Staging)
1. 在您的電腦開發新功能（如多語系）。
2. 在本地運行 Docker 測試，確保一切功能正常且不會導致 `config.inf` 格式崩潰。

### 第二步：資料庫遷移 (Migrations)
如果您新增了欄位（例如在 `User` 表新增 `language` 欄位），建議引入 `Flask-Migrate`：
- 您在開發環境執行 `flask db migrate` 生成遷移檔。
- 在 VPS 上執行 `flask db upgrade`，它會自動修改資料庫結構完成升級，而不會清空裡面的舊資料。

### 第三步：遠端部署 (Deployment)
在 VPS 上執行以下指令即可安全更新：
```bash
# 1. 備份保險（更新前手動備份一次資料庫資料夾）
cp -r ./database ./database_backup_v1

# 2. 拉取最新代碼 (如果是用 Git)
git pull origin main

# 3. 重新構立並啟動（Docker 會自動偵測變更並僅更換修改的部分）
docker-compose up -d --build
```

---

## 3. 三重保險機制 (Triple Safety)
1. **本地備份**：更新前手動 `cp` 備份資料庫檔案。
2. **容器隔離**：利用 Docker Volumes 確保 `up -d` 不會刪除資料。
3. **雲端後援**：您剛實施的 **Google Drive 背景備份**。萬一伺服器故障或手動操作失誤，直接從 Google Drive 下載最新的 `.sqlite` 檔案覆蓋回 `database/` 即可復活。

---

## 4. 針對「方便性」的建議
- **自動化監控**：建議在 VPS 安裝 `Watchtower` 容器，它可以自動偵測您的 Docker Hub 鏡像更新並自動完成重啟歸位，達成「無人值守更新」。
- **管理小圈圈**：建議保留一組 `dev` 版本的 VPS，先在那邊開發測試，確定沒事再推到正式客戶的 VPS。
