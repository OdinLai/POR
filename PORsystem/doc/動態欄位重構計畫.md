# 動態欄位配置 (Dynamic Columns) 實施方案

目前的系統架構在「資料庫欄位」、「後端邏輯判斷」以及「前端 HTML 模板」中均硬編碼了四個固定階段（訂貨、到貨、製作、出庫）。為了實現您提出的「動態適配更多或更少欄位」的需求，我建議進行以下重構。

## 使用者評論與決策確認

> [!IMPORTANT]
> **資料庫結構變動**：此改動需要將原本分散的日期欄位（如 `order_date`, `arrival_date`）整合進一個 JSON 欄位中。雖然您可以更靈活地增減欄位，但這屬於破壞性變更（Breaking Change），現有測試資料可能需要清空或遷移。

## 建議改進方案

將固定邏輯解耦，改為基於 `config.inf` 的驅動模式。

### 1. 配置定義 (Configuration)
在 `config.inf` 中新增欄位定義段落：
```ini
[Workflow]
# 階段 ID (內部邏輯用), 顯示名稱 (繁體中文)
stages = ORDER:訂貨, ARRIVAL:到貨, PRODUCTION:製作, DELIVERY:出庫
# 初始階段 ID
initial_stage = ORDER
```

---

### 2. 資料庫模型重構 (Models)
#### [MODIFY] [models.py](file:///d:/PythonWorking/POR/PORsystem/models.py)
- **SignboardItem**: 刪除 `order_date`, `arrival_date` 等欄位，改用一個 `stage_dates` (JSON 格式) 儲存所有時間戳。
- **Permission**: 刪除 `can_transfer_arrival` 等欄位，改用 `stage_perms` (JSON 格式) 儲存權限矩陣。

---

### 3. 後端邏輯通用化 (Backend Logic)
#### [MODIFY] [main.py](file:///d:/PythonWorking/POR/PORsystem/main.py)
- **初始化**：啟動時讀取 `config.inf` 中的 `stages` 並存入全域變數或 app context。
- **轉移邏輯**：在 `/api/transfer` 中不再使用 `if target == 'ARRIVAL'`，而是根據設定好的 `stages` 順序進行通用判斷。
- **權限檢查**：根據 `stage_perms` JSON 中的對應 Key 進行檢查。

---

### 4. 前端模板動態生成 (Templates)
#### [MODIFY] [show.html](file:///d:/PythonWorking/POR/PORsystem/templates/show.html)
#### [MODIFY] [manage.html](file:///d:/PythonWorking/POR/PORsystem/templates/manage.html)
- 使用 Jinja2 的 `for` 迴圈遍歷 `stages` 配置，動態生成表格標題、資料單元格以及管理按鈕。

---

## 驗證計畫

### 自動化測試
1. **設定檔測試**：修改 `config.inf` 增加一個新階段（例如：`TEST:測試階段`），重啟系統後觀察看板是否正確顯示 5 欄。
2. **流程迴圈測試**：測試從第一階段轉移至最後一個新增階段的完整邏輯。

### 手動驗證
1. **UI 適配檢查**：觀察在 Kiosk 模式下，當欄位增加到 5 欄或 6 欄時，寬度是否依然平均分配且排版整齊。
2. **權限矩陣檢查**：在「用戶管理」頁面檢查是否能正確為新增加的階段分配權限。
